# GUIDELINES_DRIZZLE_MIGRATION_LINT.md

## Core Philosophy — a trustworthy linter over drizzle-kit's own artifacts

`drizzle-migration-lint` reads the artifacts drizzle-kit already generates
(migration SQL + snapshots) and flags operations that are unsafe to run against
a live production database, with the safe alternative spelled out in every
message. It is the missing "strong_migrations for Drizzle".

Two properties define it, and every decision must protect them:

1. **Trustworthy.** It is READ-ONLY and offline by default — it never runs a
   migration or imports a user's config. The *sole* exception is opt-in
   `--db-url` table-size introspection: a single read-only `SELECT` the user
   must explicitly enable, never touched otherwise. A linter that people disable
   because it is noisy or dangerous is worse than no linter. Low false-positive
   rate is a feature, not a nicety.
2. **Drizzle-aware.** It reads drizzle-kit's **snapshots**, so it works from
   semantic diffs, not raw SQL text. That is the whole differentiator versus a
   generic SQL linter (squawk): it knows which tables are new in a migration.

Be honest about scope. Postgres is first-class (real statement-level analysis);
sqlite and mysql get the structural rules and are experimental. The README's
comparison table (vs squawk, Atlas, `drizzle-kit check`) must stay accurate and
non-adversarial. Never imply affiliation with the Drizzle team.

## 1. Architecture (never break these)

- **Two layers.** (1) *Structural* — diff consecutive snapshots into semantic
  `DiffOp`s; all dialects. (2) *Statement* — parse Postgres migration SQL with
  the real Postgres parser (`libpg-query`); Postgres only.
- **Dual-format behind ONE internal model.** The legacy journal layout
  (`meta/_journal.json` + `NNNN_*.sql`; pg v7 / sqlite v6 / mysql v5 snapshots)
  and the v1 folder layout (`<timestamp>_<name>/`; pg v8 flat-DDL snapshots with
  a `prevIds` DAG, dialect literal `postgres`) are normalized by the format
  readers into the same `Snapshot` / `Migration` types. **The engine and rules
  must never branch on artifact format** — if they need to, the normalizer is
  incomplete.
- **The new-table exemption is sacred.** Any operation on a table created in the
  same migration is exempt (nobody can be reading it yet). First migration →
  everything new → zero findings. When a snapshot is missing, fall back to
  harvesting `CREATE TABLE` names from the SQL; **never bridge across a snapshot
  gap** (it misattributes cumulative changes).
- **`lint()` is async only** to await the one-time WASM parser init. Rules stay
  synchronous over a pre-parsed AST.

## 2. Non-negotiable quality gates

- **100% coverage** — statements, branches, functions, lines (c8
  `--check-coverage`). No drops. `c8 ignore` only for a genuinely unreachable
  defensive branch, with a comment explaining why it can't be hit.
- **Cognitive complexity ≤ 15** per function (SonarJS). Extract helpers; never
  weaken a rule to fit an arbitrary shape. CI posts a per-PR complexity report.
- **`libpg-query` is the ONLY runtime dependency.** Lazy-loaded via dynamic
  import, Postgres-only, so sqlite/mysql users pay nothing at runtime. The CLI
  uses `node:util` `parseArgs` — zero dependencies. Do not add a second runtime
  dep without extraordinary justification.
- **Deterministic fixtures.** Regenerated by BOTH pinned drizzle-kit versions;
  `npm run fixtures:regen && git diff --exit-code test/fixtures` must be clean.
  Hand-edited fixtures are marked `"mode": "handmade"` and skipped by regen.
- **Never execute user code.** `drizzle.config.*` is regex-scanned, never
  imported (no jiti/ts-node). No database connection by default — the only
  exception is opt-in `--db-url` size introspection (read-only, one query, `pg`
  as an optional peer). It must never write, run migrations, or connect unasked.

## 3. Rule design

- Every `Finding` carries: rule id, severity, a clear message, the **safe
  alternative** (`suggestion`), file + line, and a docs anchor into
  `docs/rules.md`. If you add a rule, add its anchored section too.
- **Severity policy:** `error` = will lock or rewrite the table or fail under
  load (Postgres lock-safety); `warn` = rolling-deploy hazard (drops, renames,
  truncate). Default `--fail-on error`.
- The **new-table exemption applies to every rule.**
- **Postgres AST — proto3 semantics:** absent fields mean false/default, so
  test field PRESENCE, never `=== false`. A golden-AST test pins the parsed JSON
  for a reference statement so an upstream field rename fails loudly.
- The **safe-widening whitelist** is its own matrix-tested module. Note:
  drizzle-kit emits `USING "col"::type` on *every* `ALTER COLUMN ... TYPE` —
  detect and ignore that mechanical cast, or safe widenings false-positive.
- **`now()` / `CURRENT_TIMESTAMP` are STABLE and must NOT flag** as volatile
  defaults (this corrects a widespread folk rule). Keep the volatile-function
  list executable through fixtures.

## 4. Format & version drift

- Pin artifact shapes to **observed JSON from real generated files**, not to
  drizzle-kit's zod schemas (which have `.default({})` fields that real files
  omit). Re-verify on every drizzle-kit major.
- The **drift-probe** CI job regenerates fixtures against the latest and
  prerelease drizzle-kit dist-tags and fails on divergence — the early warning
  for an upstream format change (e.g. a new snapshot version). It is
  non-blocking (`continue-on-error`).

## 5. Reporters, CLI, scoping

- **Reporters:** `pretty` (human; `util.styleText` feature-detected, `NO_COLOR`
  respected), `json` (stable `{ version: 1, findings, diagnostics, summary }` —
  bump `version` on any breaking shape change), `github` (annotations + job
  summary; auto-default under `$GITHUB_ACTIONS`).
- **Exit codes:** 0 clean (or below `--fail-on`), 1 findings, 2
  usage/environment error. Diagnostics (missing snapshot, broken chain, parallel
  branches) are reported, never fatal — lint best-effort and keep going.
- **Suppressions:** `-- drizzle-migration-lint:disable-next-statement|disable-file
  <rule-id>[,...] [reason]`. Suppressed findings stay visible in summary counts.
- **Scoping:** `--since <git-ref>` compares migration *sets* at a ref (fail-safe
  on squashed/rewritten history → lint everything); `--all`; a config
  `baseline`. The `baseline` command writes only the `baseline` key.

## 6. Non-goals (keep them honest in the README)

- Not a query linter (that is `eslint-plugin-drizzle`'s job).
- Never executes migrations or config, and never connects to a database unless
  you opt in with `--db-url` (read-only size introspection).
- No auto-rewrite — every finding gives the safe recipe; applying it stays a
  human decision.
- No runtime hooks. Live table-size introspection exists but is strictly opt-in.

## 7. Testing discipline

- `node --test` + `ts-node`. Unit tests per module, a generic fixture harness
  (compare engine output to `expected.json`), and a spawn-based CLI e2e test.
- **Mutation testing (Stryker) is LOCAL-ONLY and must never be wired into CI.**
  It is an occasional, targeted audit (scope with `STRYKER_MUTATE`), not a
  per-change gate. The 100% coverage gate is the CI bar.

## 8. Release

- Push a `v*` tag → CI publishes with provenance (`npm publish --provenance
  --access public`, `id-token: write`). Publishing happens **in CI only** —
  provenance requires GitHub Actions OIDC and cannot be produced locally.
- Update `CHANGELOG.md` per release; `test:cov` must be green (the release
  workflow re-runs it before publishing).
- The `NPM_TOKEN` secret must be a token that can publish *this* package. A
  first publish of a new package needs a create-capable token — one scoped only
  to pre-existing packages will `E403` on the initial `PUT`.

## 9. When in doubt

Ask: *does this keep the tool trustworthy (read-only, honest, low false
positive), and does it hold every gate (100% coverage, complexity ≤ 15, one
runtime dep, deterministic fixtures)?* If not, redesign until the answer is yes.
Prefer a smaller, correct, well-explained rule over a broad, noisy one.

Follow these assumptions in **every** file you generate or modify. This is the
project constitution.
